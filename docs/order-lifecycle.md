# 訂單生命週期與情境清單（草案，逐項定案後才動工）

> [rental-rules.md](./rental-rules.md) 管**靜態規則**（上限、押金、天數）；本文件管**動態情境**——
> 一張訂單從送出到結束的每個節點：誰觸發、何時發生、對「佔用／押金／通知／帳號狀態」的連鎖影響。
> 標 ★ = 尚未決定，需要 Bernard 定案；標 ⚠ = 已知落差（現況與預期不符）。
> 這是階段 1.5 驗收與階段 2 後台的前置文件：**情境沒定案，後台沒得做**。

## 現有狀態機（資料庫 orders.status）

```
pending（待繳押金）──繳押金[admin]──▶ in-progress（租借中）──歸還[admin]──▶ returned（已歸還）
    │                                      │
    │ 24 工作時未繳[pg_cron 自動]           │ 過有效歸還期限 19:01[pg_cron 自動]
    ▼                                      ▼
canceled（取消）                        overdue（逾期）──補歸還+罰款[admin]──▶ returned
```

佔用規則（已實作）：`pending / in-progress / overdue` 佔用庫存與空間；`returned / canceled` 釋放。

## 情境清單

### 1. 借了沒繳押金 ✅ 已定案（2026-07-14，已實作）
- **規則**：pending 超過 **24 工作時**未繳押金 → 自動改 `canceled`、釋放佔用、發通知。
  - 工作時排除**週六日＋臨時公休日**（系學會活動等不營業日，公休整天暫停倒數）。
  - 臨時公休日存 `closed_dates` 表，過渡期由系學會在 Studio 手動加日期。
- **實作**（`supabase/auto-cancel.sql`）：pg_cron 每 10 分鐘跑 `cancel_expired_pending()`——
  資料庫 status 真的改 `canceled`（方案 b），同時插入「訂單已取消」通知。
  前端 `utils/timeUtils.ts` 有同邏輯的顯示用判定（`displayOrderStatus`），
  ProfilePage 與 OrderPage 共用同一份，掃描間隔內 UI 也即時正確。
- **不記次數**：送單不來繳只影響學生自身權益（佔用會釋放，不影響他人），
  不入帳號 6 級制；canceled 單留在資料庫，需要統計累犯時隨時可查。

### 2. 學生自行取消訂單 ✅ 已定案（2026-07-14）
- **規則**：學生**一律不能自行取消**（前台無取消功能）。
  - `pending` 沒繳押金：不去學會即可，時間到走情境 1 自動作廢（唯一的取消途徑）。
  - `in-progress` 要取消：**必須到學會現場處理**（涉及退押金），前台不做 → 留給階段 2 後台 admin。

### 3. 學生修改已送出的訂單 ✅ 已定案（2026-07-14），一例外待討論
- **規則**：學生**不能修改**已送出的訂單（換日期／加減品項都不行）。
- **唯一路徑**：需變更 → 到學會後台由 admin 更改。前台與後台都不做「學生自助編輯」，
  後台只需 admin 端的修改能力（併入情境 9）。
- **⏸ 待討論的例外——大量訂單事後加設備**：大量單成立後可能臨時要再加設備。
  由 admin 後台代改，還是**允許原下單學生自己在前台加**？待 Bernard 定案。

### 4. 繳押金／交接（pending → in-progress）✅ 已定案（2026-07-14，待階段 2 後台實作）
- **規則**：24 工作時內繳押金即可（無「最早可繳」限制），**繳的當下直接取件**——
  不需要「已繳未取件」中間態。後台只要一顆「確認收押金」按鈕（pending → in-progress）。
- 交接拍照記錄不進系統（人工流程）。

### 5. 歸還／部分歸還（in-progress → returned）◐ 主結構定案，兩處待討論（2026-07-14）
- **現況**：Studio 手動改；部分歸還**完全記錄不到**（status 整張單一個值，品項無歸還欄位）。
- **核心區分（2026-07-14 定案）**：**部分歸還 ≠ 部分延期**，差別在**有沒有動用延期權**：
  - **部分歸還**＝期限**內**提早歸還部分設備。剩餘品項拆成子單 `-1`，子單**仍保有延期效用**
    （延期權沒被動用，`has_extended` 沿用原單、通常仍 false → 之後還能延一次）。歸還日不變。
  - **部分延期**＝期限（後／到期）想延**部分**設備。延的那批**用掉延期權**（延期本就是延期），
    延期子單 `has_extended = true`、不能再延；沒延的品項按原歸還日還。
  - （兩者可交叉描述，但拆單時以「這次有沒有 extend」決定子單的 `has_extended`。）
- **拆單機制（定案）**：分兩批的品項——一批留／關原單，另一批拆成**子單**，續走逾期／罰款／延期／停權流程，
  狀態機「一張單一個狀態」不變量不動。
  1. **單號 ✅**：子單 `20260401-1`（原單 `20260401` ＋後綴），orders 加 `parent_order_id` 供追溯。
     **全部品項一起延、沒有任何一項被擋** → **不拆單**，原單 `20260401` 直接延長 end_date（走情境 8）。
  2. **押金拆分**：
     - **小單 ✅**：未還品項押金小計＝子單 `deposit_total`（小單子單押金**必不超過 5000**，cap 不會綁到），
       差額退還學生現金。差額**存在原單新欄位 `refunded`** 供後台追溯（原單已結案，新單看子單押金即可，
       但退多少留記錄以防查帳）。
     - **大單 ⏸ 待討論**：大單一般不會出現此情況——先留著，Bernard 晚點再給拆分規則（RPC 先擋大單）。
  3. **延期權 ✅**：拆單時 `has_extended` 依情境定——
     - 部分**歸還**（提早）：子單**沿用**原單 `has_extended`（沒動用延期）。
     - 部分**延期**：延期子單設 `has_extended = true`（用掉延期權）。
  4. **歸還日 ✅**：
     - 部分**歸還**（提早）：子單 end_date **不變**（沿用原歸還日），只是已還部分關掉。
     - 部分**延期**：子單 end_date = 原歸還日 ＋ 延期天數（**上限 7 天**，同一般延期）；未延品項在原歸還日還。
  5. **權限與前台介面 ★ 待思考**：拆單本身僅 admin。但既然部分歸還＝部分延期，
     考慮**讓學生在前台就選「哪些設備要延期、哪些在原歸還日還」**，到歸還日再與學會一起處理子單。
     此前台介面要不要做、怎麼做，Bernard 待思考後定案。
- **6. 損壞賠償 ⏸ 待討論**：文件無既有規則。往例做法＝**賠錢／買新品補上／扣押金**，
  **不因損壞停權**（不歸還才嚴重）。輕微／嚴重的分級與金額認定待正式定案。

### 6. 逾期（in-progress → overdue）✅ 已定案（2026-07-14，已實作）
- **判定**：有效歸還期限 = 歸還日 19:00（系學會最後營業時間），超過即逾期（19:01 起算）；
  歸還日撞週末／臨時公休（`closed_dates`）自動順延到下一個營業日——公休還不了不怪學生。
- **實作**（`supabase/auto-overdue.sql`）：pg_cron 每 5 分鐘跑 `mark_overdue()`
  自動改 `overdue` 並發逾期通知（含罰款提醒），學會看後台就知道誰沒還。
- **罰款**：第 1 日 100 元、每日翻倍**累計**（第 3 日欠 100+200+400=700），上限押金總額。
  逾期中由前端即時試算顯示（列表＋詳情頁，`timeUtils.overduePenalty`），**不入庫**；
  歸還時最終金額由系學會確認後寫入 `orders.penalty_total`（欄位已加，介面留給階段 2 後台）。
- **假日不計**（2026-07-14 隨情境 7 一併定案）：罰款天數與停權門檻的「日」都**不含**
  週末與臨時公休（與 rental-rules §10、官方 Google 文件一致）；曾短暫暫定含假日，已改回。

### 7. 逾期滿 6 日／罰款達上限 → 未完成清潔歸還 ✅ 已定案（2026-07-14，已實作）
- **官方依據**（Google 文件原文）：**單筆**訂單逾期達 6 日（**不含假日**）＝未完成清潔歸還
  → 不可再使用預約系統、系辦不予核章、承擔費用。畢業不核章屬行政面，不進系統。
- **重要澄清**：官方文件**沒有**「帳號 6 級制／累計逾期天數」——分級是 app 自己的顯示設計。
  停權依據只看單筆；`students.overdue_days` 欄位保留未用。
- **實作**（`supabase/account-suspension.sql`）：
  - pg_cron 每小時跑 `suspend_overdue_accounts()`：任一 overdue 單滿 6 個營業日
    → `account_level = 5`＋停權通知。**sticky**：歸還後仍鎖，系學會人工解（Studio 改回 0）。
  - `submit_orders` 開頭擋 `account_level >= 5`（**擋送單、不擋登入**——學生仍可查通知與欠款）。
  - 標籤第 1–6 級**即時計算不入庫**（Profile info，`ProfileDataSection`）：
    第 1 級＝無逾期（Good Standing）、第 2–5 級＝當前最嚴重單逾期 1–4 營業日（黃色警示）、
    第 6 級＝逾期 5 日（紅色最後警告）、滿 6 日顯示停權。

### 8. 延期
- **已實作**：僅乙次、1–7 天、僅 in-progress，資料庫端驗證；延期直接生效（無需審核）。
- **撞期檢查 ✅（2026-07-14 定案並實作）**：`extend_my_order` 逐品項檢查延長區間
  （原歸還日+1 ～ 新歸還日）與其他生效訂單的衝突——設備看數量、空間看佔用，
  **全部品項都能延才放行**，否則列出撞期品項名稱拒絕。
  部分延期（A 能延 B 不能）＝走情境 5 的拆子單，屬階段 2 後台 admin 功能。
- **前三天提出 ✅ 已定案並實作（2026-07-14）**：官方原有規則，**程式強制**——
  延期須在原歸還日前三天（含）提出（申請當日 ≤ 歸還日 − 3 天），過期即不可延、訂單照常結束；
  仍想借須重新下單走先來後到。`extend_my_order` RPC 依日期擋（伺服器防線）、
  ProfilePage 延期鈕以 `timeUtils.isWithinExtendWindow` 同步禁用（UX，有 vitest 邊界測試）。
- 延期直接生效、無需 admin 同意（現行學生自助延期照舊）；admin 代延期不受此期限限制（情境 9）。

### 9. 管理員代操作 ✅ 已定案（2026-07-14，待階段 2 後台實作）
- **後台功能清單**：
  1. **確認收押金**：手動改 `pending → in-progress`（情境 4 的那顆按鈕）。
  2. **協助延期**：代學生延期，不受前台「僅乙次」限制。
  3. **拆分訂單**：涵蓋情境 5 部分歸還與情境 8 部分延期（A 能延 B 不能）的拆子單。
  4. **帳號狀態調整**：改 `account_level`（含老師通融解除停權、手動升降級）。
  5. **訂單全覽／查詢**：誰借（姓名／學號）、借什麼、借到何時、狀態篩選（其他功能的基礎）。
  6. **歸還標記**：`in-progress → returned`，並確認逾期罰款最終金額寫入 `orders.penalty_total`。
  7. **手動建單**：電話／現場預約，admin 代學生下單。
  8. **臨時公休日／寒暑假封鎖維護**：`closed_dates`（情境 1／6 倒數暫停）與
     `rental_blackouts`（情境 11-a）的區間管理（過渡期在 Studio 手動，後台化為之後優化）。
  9. **庫存管理**：設備新增／下架（`is_active`）／調整數量（`stock_quantity`）——見情境 11。
- **原則（2026-07-14 定案）**：學會一切操作都走**後台**，資料庫只是資料存放層；
  過渡期未做的後台功能才暫用 Studio 頂替。凡「學會日常會做的事」都應有後台入口。
- **訂單編輯政策（2026-07-14 定案）**：**不做「自由編輯任意欄位」的表單**。分兩類：
  - **軟性描述**（`reason`／`class_name`／`teacher`）：不碰不變量 → **可就地 inline 編輯**。
  - **硬性資料**（日期／品項／數量／狀態／押金）：碰佔用／押金／流水號／狀態機不變量
    → **只能走專用動作按鈕**（各自一顆有把關的 RPC，如收押金／歸還／代延期／拆單），絕不自由改。
  - 一句話：不影響不變量的可自由改，一碰佔用／押金／狀態／日期就必走專用動作。

### 10. 助教直借（staff）✅ 已定案（2026-07-14）
- **規則**：走一般前台流程選借，但**免押金、送出即 `in-progress`**（跳過 pending 繳押金）。
- **仍守先來後到**：一樣佔用庫存／空間，不能插隊搶已被佔用的資源；
  唯有 admin 從後台協調時才可調整。歸還流程與學生相同。

### 11-a. 寒暑假封鎖（新制度）✅ 已定案並實作（2026-07-14）
- **規則**：寒假與暑假期間**一律不開放學生租借**；具體起訖日由 admin 決定；
  系辦超級帳號（admin）不受限制。
- **實作**（`supabase/rental-blackouts.sql`；與 `closed_dates` 分開——公休只暫停倒數，封鎖是擋新單）：
  - 表 `rental_blackouts (id, start_date, end_date, reason)`＋RLS（公開讀、admin 寫），過渡期 admin 在 Studio 維護。
  - `submit_orders` 檢查：`user_role() = 'student'` 且租借區間與任一封鎖區間重疊 → 拒絕；admin／staff 不擋。
  - 前端日曆**直接擋**封鎖區間 ✅（2026-07-14 實作）：`Calendar` 讀 `fetchBlackouts()`，
    封鎖日期灰化不可點、跨封鎖的起訖選擇也擋；admin／staff 不載入不受限。server 端仍為防線。

### 11. 設備異動 vs 進行中訂單 ✅ 已定案（2026-07-14）
- **規則**：設備異動走**後台庫存管理**（情境 9.9）——新增就加 `stock_quantity`，報廢／遺失就減；
  下架改 `is_active`。（依「學會一切走後台」原則，不直接動資料庫。）
- **衝突處理**：若在庫數減到低於「已被預約的量」（例如學生已借出，期間有損壞導致總數下降，
  而該生無法改單）——**不做系統警示或自動處理**，一律由學會人工協調。後台不需為此寫邏輯。

## 建議的討論順序

1 → 6 → 7（同一條「單子死掉」的鏈，決定「誰改狀態」是共同前提：排程 or 查詢排除 or 人工）
→ 4 → 5（後台日常操作）→ 2 → 3（學生自助能力範圍）→ 8 ⚠（延期撞期是唯一的規則漏洞）
→ 9 → 10 → 11（後台功能清單自然浮現）。
